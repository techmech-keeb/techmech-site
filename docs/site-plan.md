# Techmech keys Webサイト構築計画

## 目的

Techmech keysの取扱商品を紹介し、個人Makerとしての活動、商品情報、Firmware、販売サイト、問い合わせ先への導線を一箇所に集約する。名刺やイベント、SNS、知人への紹介時に共有する代表URLとして機能させ、商品の魅力と購入・利用までの流れをわかりやすく伝える。

## 前提

- GitHubで管理する本リポジトリをWebサイトのソースとして利用する。
- ホスティング先はNetlifyを想定する。
- 初期段階では、軽量で保守しやすい静的サイトとして構築する。
- 商品数、販売チャネル、Firmwareの配布方法は今後増減する可能性があるため、情報更新のしやすさを重視する。

## 1. 個人Makerとしてのドメイン活用アプローチ

### 1.1 ブランドの公式ハブとして活用

独自ドメインをTechmech keysの公式情報の集約先として利用する。SNS、販売サイト、GitHub、Firmware配布先などに情報が分散していても、最終的な入口をこのWebサイトに統一する。

主な役割は以下の通り。

- 取扱商品の一覧と特徴の紹介
- 商品ごとの購入先への誘導
- Firmware、ビルドガイド、マニュアルへの誘導
- イベント出展時や名刺交換時の紹介先
- Maker本人の活動実績や制作思想の発信

### 1.2 商品カタログとして活用

商品ごとにページまたはセクションを用意し、以下の情報を整理する。

- 商品名
- 商品写真またはレンダリング画像
- コンセプト
- 特徴
- 対応Firmware
- 販売ステータス
- 購入リンク
- ビルドガイド、利用ガイド、サポート情報

初期段階では1ページ内に商品カードを並べ、商品数が増えた段階で個別商品ページへ拡張する構成が適している。

### 1.3 信頼形成のためのポートフォリオとして活用

個人Makerの場合、商品のスペックだけでなく「誰が、どのような意図で作っているか」が購入判断に影響する。Webサイトには以下の要素を含めるとよい。

- Techmech keysの制作方針
- キーボード、キーキャップ、基板、ケースなどへのこだわり
- 過去の制作物や試作履歴
- イベント参加履歴
- SNS、GitHub、販売サイトへのリンク

### 1.4 名刺・イベント用ランディングページとして活用

名刺、ショップカード、イベントPOP、QRコードからアクセスされることを想定し、トップページのファーストビューで以下を明確に伝える。

- Techmech keysが何を扱っているか
- 代表的な商品の魅力
- 購入・Firmware・SNSへの主要導線
- 最新情報の確認先

スマートフォンから閲覧されるケースが多いため、モバイル表示を優先して設計する。

## 2. Webサイト構築技術の方針

### 2.1 推奨構成

初期構築では、静的サイトジェネレーターまたは軽量なフロントエンド構成を推奨する。

候補は以下の通り。

| 技術 | 特徴 | 適性 |
| --- | --- | --- |
| Astro | 静的サイト向き。Markdownやコンポーネント管理がしやすい。Netlifyとの相性もよい。 | 推奨 |
| Vite + React | UIの自由度が高い。将来的に動的要素を増やしやすい。 | 中規模以上に拡張する場合に適する |
| Next.js | 高機能だが、初期の静的商品紹介サイトとしてはやや過剰。 | ブログ、CMS、動的機能を強く使う場合に検討 |
| 素のHTML/CSS | 最小構成で軽い。依存が少ない。 | 超小規模な初期版には適するが、継続更新時の管理性は低い |

更新頻度が多くなく、ページ数も限定的で、デザインにこだわりたい前提では「静的に配信できること」「CSSやアニメーションを制御しやすいこと」「過剰な運用負荷を持ち込まないこと」を重視する。現時点ではAstroを第一候補とするが、制作スタイルによっては素のHTML/CSSやEleventyも有力である。

### 2.2 評価軸

技術選定では以下の観点で比較する。

- デザイン自由度: レイアウト、アニメーション、写真・ビジュアル表現を作り込みやすいか。
- 保守性: 商品情報、リンク、画像、文言を後から更新しやすいか。
- 実装コスト: 初期構築に必要な学習量や設定量が大きすぎないか。
- Netlify適性: 静的ファイルとしてビルドし、Netlifyで安定して配信しやすいか。
- 将来拡張: 商品ページ、制作記録、FAQ、多言語化などに拡張しやすいか。
- 依存の少なさ: 小規模サイトに対してフレームワークやビルド設定が過剰でないか。

### 2.3 技術候補の比較

| 技術 | 向いているケース | メリット | 懸念点 | 本サイトでの評価 |
| --- | --- | --- | --- | --- |
| Astro | 静的な商品紹介、少量の記事、デザイン重視のブランドサイト | HTML中心で軽量。Markdown/コンポーネント管理がしやすい。必要な部分だけReact等を使える。 | Astroの作法を覚える必要がある。 | 最有力。小規模から拡張までバランスがよい。 |
| Eleventy | Markdown中心の静的サイト、シンプルな情報サイト | 依存が比較的軽く、静的HTML生成に強い。テンプレートの自由度が高い。 | UIコンポーネント開発体験はAstroより素朴。作り込みは設計力が必要。 | 有力。更新頻度が少なく、文書中心なら適する。 |
| Vite + React | 動きのあるUI、インタラクション重視のサイト | コンポーネント表現が強く、デザイン実装の自由度が高い。 | 静的な数ページにはややアプリ寄り。JS量が増えやすい。 | デザイン演出や動的UIを強めるなら候補。 |
| Next.js | CMS連携、多言語、将来的なWebアプリ化 | 機能が豊富で拡張性が高い。 | 小規模な静的サイトには過剰になりやすい。運用・設定の考慮点が増える。 | 現段階では優先度低め。 |
| 素のHTML/CSS/JavaScript | 1〜数ページの固定サイト、完全に手作業で作るLP | 依存が少なく、表示が速く、デザインを直接制御できる。 | 共通部品、商品追加、リンク更新が手作業になりやすい。 | 初期LPだけなら有力。継続管理では注意。 |
| Webflow等のノーコード | デザインを素早く作りたい、コード管理を重視しない | ビジュアル制作が速い。アニメーションやレスポンシブ調整がしやすい。 | GitHub中心の管理やNetlify運用とは相性が弱い。エクスポート後の保守に注意。 | このリポジトリで管理する前提では優先度低め。 |
| SvelteKit | 軽量なUIと動きを両立したい | コンポーネントが軽く、表現力が高い。 | 小規模静的サイトでは学習・設定がやや増える。 | UI実験を重視する場合の候補。 |

### 2.4 前提別の推奨

更新頻度が少なく、ページ数も限定的な場合は、以下の判断が現実的である。

- 最小の1ページLPとして早く公開したい場合: 素のHTML/CSS/JavaScript。
- デザインを作り込みつつ、商品情報やリンクを管理しやすくしたい場合: Astro。
- Markdown中心で、制作記録やドキュメントを少しずつ増やしたい場合: AstroまたはEleventy。
- Reactコンポーネントや複雑なアニメーションを多用したい場合: Vite + React。
- CMS、多言語、ログイン、API連携などを将来的に重視する場合: Next.js。

本サイトの初期条件では、ページ数が限定的でも「商品カード」「Firmware導線」「販売先導線」「今後の商品追加」を考えると、素のHTML/CSSだけでは後から情報更新が煩雑になりやすい。そのため、静的サイトとして軽量に公開でき、デザイン実装と情報管理のバランスがよいAstroを推奨する。

### 2.5 デザイン重視の場合の実装方針

デザインにこだわる場合でも、技術は過剰に複雑にせず、以下の方針で進める。

- まずFigmaなどでトップページの見た目と情報優先度を決める。
- Astroでセクション単位のコンポーネントを作成する。
- CSSはTailwind CSSまたは通常のCSS Modules/グローバルCSSから選ぶ。
- ブランドカラー、余白、角丸、タイポグラフィ、ボタン、カードをデザイントークンとして整理する。
- アニメーションはページ表示速度と可読性を損なわない範囲に限定する。
- 商品情報はMarkdown、JSON、TypeScriptのデータファイルなどで分離し、デザインと内容を分けて管理する。

デザインの自由度を最優先するなら通常のCSSまたはCSS Modules、実装速度と統一感を重視するならTailwind CSSが候補になる。どちらの場合も、初期段階ではUIライブラリを全面採用するより、Techmech keysらしい見た目を独自に作る方がブランドサイトとして適している。

### 2.6 Astroを採用する理由

- 静的サイトとして高速に配信できる。
- Netlifyへデプロイしやすい。
- 商品情報や記事をMarkdown/MDXで管理しやすい。
- ReactなどのUIライブラリを必要な箇所だけ追加できる。
- 個人Makerサイトとして、シンプルな構成から段階的に拡張できる。
- ページ数が少ない場合でも、共通レイアウトや商品カードを部品化できる。
- デザイン重視のLPとして作り込みながら、将来の商品追加にも対応しやすい。

### 2.8 Netlify向け設定方針

Netlifyでは以下の構成を想定する。

- Build command: `npm run build`
- Publish directory: `dist`
- Node.js version: LTS系を指定
- 必要に応じて`netlify.toml`でビルド設定を管理

### 2.9 Netlifyとプライベートリポジトリの接続

Netlifyへの接続は、プライベートリポジトリでも問題ない。GitHub連携時にNetlifyへ対象リポジトリへのアクセス権を付与すれば、Netlify側でリポジトリを取得してビルドできる。サイトのソースコードや`docs/`をGitHub上で公開したくない場合は、リポジトリをprivateにしたままNetlifyへ接続する運用が適している。

プライベートリポジトリで運用する場合の利点は以下である。

- `docs/`に置いた計画書や運用メモがGitHub上で一般公開されない。
- 公開前の商品ページ、画像、文言を外部に見せずに準備できる。
- Netlify Deploy Previewを使って、公開前に表示確認しやすい。
- 公開されるのはNetlifyのPublish directory、つまり通常は`dist`のビルド結果に限定できる。

注意点は以下である。

- Netlifyにはリポジトリの読み取り権限を付与するため、信頼できるNetlifyチーム・アカウントに接続する。
- GitHub Organization配下のprivateリポジトリでは、プランや権限設定の影響を受ける可能性があるため、接続前にNetlify側のチーム設定を確認する。
- APIキー、トークン、個人情報などの秘匿情報は、privateリポジトリであってもコミットしない。必要な値はNetlifyの環境変数で管理する。
- `public/`や`src/pages/`に置いた内容は、ビルド後に公開される可能性があるため、privateリポジトリでも公開前チェックは必要である。

結論として、このサイトはGitHubリポジトリをprivateにしたままNetlifyへ接続する運用で問題ない。むしろ初期段階では、計画書、下書き、公開前のデザインを同じリポジトリで管理しやすくなるため、privateリポジトリ運用の方が安全である。公開後もソースを公開する理由がなければ、privateのままでよい。

### 2.10 サイト計画と公開コンテンツの管理方針

サイトプラン、要件メモ、運用メモなどの管理用ドキュメントと、Netlifyで公開するWebサイト本体は、同じリポジトリで管理できる。ただし、よりよい運用としては「公開してよい情報」と「内部管理用の情報」を分け、公開リポジトリに置く内容を意識的に制限する。

Techmech keysのような個人Makerサイトでは、初期は小さく始めつつ、商品情報、販売導線、Firmware導線、イベント告知を継続的に更新する可能性がある。そのため、以下のハイブリッド管理を推奨する。

### 2.11 別リポジトリに分ける必要性

結論として、公開する情報と計画等の情報を必ず別リポジトリに分ける必要はない。計画書の内容が公開されても問題ない範囲であれば、同じリポジトリの`docs/`で管理する方が、実装方針とサイトソースの対応関係を追いやすい。

一方で、以下のような情報を扱う場合は、別リポジトリまたはprivateな管理場所に分けるべきである。

- 未発表商品の仕様、価格、発売時期
- 原価、仕入れ先、製造委託先、在庫数などの事業上の内部情報
- 販売戦略、収支、イベント出展前の準備メモ
- 個人情報、住所、電話番号、購入者情報
- APIキー、Netlifyトークン、外部サービスの認証情報
- 公開前のロゴ、写真、商品画像で外部に出したくないもの

判断基準は以下の通り。

| 条件 | 推奨 | 理由 |
| --- | --- | --- |
| 計画書が公開されても問題ない | 同一リポジトリで管理 | 実装と計画の差分を一緒に追える。 |
| リポジトリをprivateで運用する | 同一リポジトリでも可 | Netlify公開対象とGitHub閲覧範囲を分けやすい。 |
| リポジトリをpublicで運用する | 公開可能な計画のみ同梱 | `docs/`もGitHub上では閲覧されるため。 |
| 内部情報や未発表情報を多く扱う | 別privateリポジトリまたは別ツール | 誤公開や履歴への残存リスクを下げる。 |
| タスクやメモが実装PRと強く結びつく | GitHub Issues/Projectsを併用 | 作業単位で管理しやすい。 |

このサイトでは、初期段階は同一リポジトリで問題ない。ただし、`docs/`に置くのは「公開されても困らない方針・設計・運用メモ」に限定する。未発表商品や販売戦略などが増えてきた段階で、privateな別リポジトリやNotion等の別管理へ分けるのが現実的である。

### 2.12 推奨する管理方法

最も扱いやすい構成は、公開サイト用リポジトリと内部管理情報を分ける方法である。

| 管理対象 | 推奨場所 | 理由 |
| --- | --- | --- |
| 公開サイトのソース | このリポジトリ | Netlify連携、レビュー、公開履歴の管理がしやすい。 |
| 公開してよい計画書 | このリポジトリの`docs/` | 実装方針や構成判断をサイトソースと一緒に追跡できる。 |
| 商品ページに使う公開情報 | `src/content/`または`src/data/` | 商品カードやページ生成に使いやすい。 |
| 画像・ロゴ・favicon | `public/`または`src/assets/` | 公開対象として管理しやすい。 |
| 未発表商品、原価、仕入れ先、販売戦略 | privateな別リポジトリまたはローカル管理 | publicリポジトリやNetlify公開物への混入を防ぐ。 |
| タスク管理、発売準備メモ | GitHub Issues/Projects、またはprivateメモ | 実装タスクと内部メモを分けやすい。 |

この方針では、このリポジトリを「公開して問題ないWebサイト本体と公開可能な設計メモの置き場」として扱う。公開したくない情報は、同じリポジトリの`docs/`ではなく、privateな場所で管理する。

### 2.13 リポジトリ内の推奨ディレクトリ分離

このリポジトリでは、当面は以下のように分けるのがよい。

```text
/
├── docs/                  # 公開してよい計画、要件、運用メモ。Netlify公開対象外。
├── public/                # 画像、faviconなど。基本的に公開対象。
├── src/
│   ├── pages/             # Webページ。公開対象。
│   ├── components/        # UI部品。ビルド結果に反映。
│   ├── content/           # 商品情報や記事。ページで使う内容は公開される。
│   └── styles/            # サイトの見た目を管理。
├── netlify.toml           # Netlifyビルド設定。
├── package.json           # 開発・ビルドコマンド。
└── README.md              # リポジトリ概要。GitHub上で閲覧される。
```

Netlifyは通常、ビルド後の`dist`ディレクトリだけを公開するため、`docs/site-plan.md`のような計画書はWebサイトのページとして明示的に取り込まない限り公開されない。ただし、GitHubリポジトリをpublicにする場合、`docs/`はGitHub上では閲覧可能になる。したがって、`docs/`にも秘匿情報は置かない。

### 2.14 ブランチと公開フロー

Netlifyで公開する前提では、以下のフローが安全である。

1. `main`ブランチを本番公開ブランチにする。
2. 変更は作業ブランチで行い、Pull Requestで確認する。
3. Netlify Deploy Previewで見た目とリンクを確認する。
4. 問題なければ`main`へマージし、本番デプロイする。
5. 商品情報、販売リンク、Firmwareリンクの変更は小さなPRに分ける。

この運用にすると、名刺やイベントで案内するURLの品質を保ちつつ、公開前にスマートフォン表示、リンク切れ、誤記を確認しやすい。

### 2.15 コンテンツ更新の管理方法

商品数が少ない初期段階では、商品情報をMarkdown、JSON、またはTypeScriptのデータファイルで管理するのがよい。ページに直接ベタ書きすると、後から商品追加や販売リンク変更を行う際に差分が追いにくくなる。

推奨は以下である。

- 商品説明、販売ステータス、購入リンク、Firmwareリンクはデータとして分離する。
- 商品画像は命名規則を決めて`public/images/products/`などに置く。
- 販売リンクやFirmwareリンクは商品データ側に集約する。
- トップページは商品データを読み込んでカード表示する。
- 終売、再販予定、イベント販売などの状態を表す項目を持たせる。

例として、商品データには以下の項目を持たせる。

```text
name
slug
summary
description
status
image
shopUrl
firmwareUrl
guideUrl
featured
```

### 2.16 公開前チェック

同じリポジトリで管理する場合は、公開前に以下を確認する。

- `public/`に未公開画像や不要ファイルが入っていないか。
- `src/pages/`に下書きページや内部メモが入っていないか。
- 商品データに未発表情報、原価、仕入れ先、個人情報が含まれていないか。
- 販売リンク、Firmwareリンク、SNSリンクが正しいか。
- NetlifyのPublish directoryが`dist`になっているか。
- スマートフォン表示で主要導線が見やすいか。

結論として、このリポジトリは「公開サイト本体 + 公開してよい計画書」を管理する場所にするのがよい。内部情報まで同じリポジトリに入れるより、公開用と内部用を分けた方が、誤公開リスクを抑えながら継続運用しやすい。

## 3. Firmware・販売サイトなどへの導線設計

### 3.1 主要導線

トップページには、少なくとも以下の導線を設置する。

- Products: 取扱商品一覧
- Firmware: Firmware配布先またはGitHub Releasesへのリンク
- Build Guide: 組み立てガイド、利用ガイド
- Shop: 販売サイト
- Contact/SNS: 問い合わせ、SNS、活動情報

### 3.2 商品ページごとの導線

各商品には、購入前後の行動を想定したリンクをまとめる。

- 購入する
- Firmwareを入手する
- ビルドガイドを見る
- 対応キー配列・仕様を見る
- サポート・問い合わせをする

### 3.3 Firmware導線の考え方

Firmwareは更新頻度があり、誤ったファイルの配布は利用者の混乱につながるため、以下を明確にする。

- 対応商品名
- 対応リビジョン
- Firmwareの更新日
- インストール手順
- 注意事項
- GitHub Releasesや公式配布先へのリンク

初期段階では、Webサイト自体にFirmwareバイナリを直接置くよりも、GitHub Releasesなどの管理された配布先へ誘導する方が安全である。

### 3.4 販売サイト導線の考え方

販売サイトへのリンクは、商品ごとに明確に設置する。販売チャネルが複数ある場合は、以下のように分ける。

- 在庫ありの購入先
- 受注・予約ページ
- 完売時の再販通知先
- イベント販売予定

販売状況は変化しやすいため、更新しやすいデータ構造で管理する。

## 4. 名刺・紹介時の集約導線としての設計

### 4.1 トップページで伝えるべき内容

名刺やQRコードから訪問したユーザーは、短時間で内容を判断する可能性が高い。トップページでは以下を優先する。

1. Techmech keysの概要
2. 代表商品のビジュアル
3. 商品の魅力を短いコピーで提示
4. 購入、Firmware、SNSへのボタン
5. 最新情報または販売状況

### 4.2 ファーストビュー案

トップページ冒頭には以下のような構成を想定する。

- キャッチコピー
- 短い説明文
- 代表商品画像
- 「商品を見る」「購入先を見る」「Firmwareを探す」の主要ボタン

例:

> Techmech keysは、メカニカルキーボードと入力体験を探求する個人Makerブランドです。日々使いたくなるキーアイテムを設計・製作しています。

### 4.3 スマートフォン優先の情報設計

名刺やイベントからの流入ではスマートフォン閲覧が中心になるため、以下を重視する。

- 主要ボタンを画面上部に配置する
- 商品写真を大きく見せる
- 長文説明よりも短い見出しと箇条書きを使う
- 外部リンクをわかりやすく分ける
- QRコードからアクセスしても迷わない構成にする
