# Techmech keys コンテンツ計画

## 目的

Techmech keysのWebサイトでは、製品の魅力を伝え、購入・サポート・Firmware・ビルドガイドへ迷わず移動できる導線を用意する。名刺、SNS、イベント、BOOTH、GitHubなどに分散している情報の入口を整理し、初めて見る人にも「何を作っているブランドか」「どの商品を見ればよいか」「どこで買えるか」「購入後にどこを見ればよいか」が伝わる状態を目指す。

## 参照元として確認した情報

### 直販ページ

- URL: <https://techmech.booth.pm/>
- 役割: Techmech keysの直販・頒布ページ。
- 現状の読み取り: 試作品の頒布が中心で、正式公開時に製品・価格・内容が変わる可能性がある旨が案内されている。
- Webサイト側での扱い: 商品ごとの購入ボタン、ショップ全体への導線、販売ステータスの確認先として使う。

### サポート情報ページ

- URL: <https://github.com/techmech-keeb/OLSK60_v2>
- 役割: OLSK60 v2のサポート・ドキュメント・Firmware情報の入口。
- 現状の読み取り: OLSK60 v2は、標準キーキャップ互換を意識した60%格子配列キーボードで、TrackPoint、GH60互換ケース、QMK/Remap対応、ホットスワップ、スタビライザー実装済みなどが特徴として整理されている。
- Webサイト側での扱い: 製品詳細、サポート、ビルドガイド、Firmware、GitHubリポジトリへの導線として使う。

## コンテンツ設計の基本方針

### 伝える順序

初見のユーザーには、詳細な仕様よりも先に「何が魅力か」「自分に関係があるか」「次に何をすればよいか」を伝える。

推奨する情報の順序は以下である。

1. Techmech keysの概要
2. 代表製品のビジュアルと短い紹介
3. 製品の特徴
4. 購入導線
5. サポート・Firmware・ビルドガイド導線
6. 詳細仕様やGitHubリンク
7. SNS・問い合わせ

### 外部サービスとの役割分担

Webサイトは情報のハブとして扱い、購入や詳細ドキュメントは既存の外部サービスへ送る。

| 種別 | 役割 | 掲載先 |
| --- | --- | --- |
| ブランド紹介 | Techmech keysの概要、制作方針 | Webサイト |
| 製品紹介 | 製品の魅力、写真、特徴、概要 | Webサイト |
| 購入 | 在庫、価格、決済 | BOOTH |
| サポート | ビルドガイド、パーツリスト、回路図、操作ガイド | GitHub |
| Firmware | Firmware配布、QMK関連情報 | GitHubまたはFirmwareリポジトリ |
| 最新情報 | 再販、イベント、試作品情報 | Webサイト、SNS、BOOTH |

## OLSK60 v2の掲載方針

OLSK60 v2は、初期サイトの中心コンテンツとして扱う。

### 伝えるべき魅力

- 標準的なキーキャップセットに対応した60%格子配列キーボードであること。
- TrackPoint搭載により、キーボードから手を離さずにポインティング操作をしやすいこと。
- GH60互換ケースに対応し、好みのケースやキーキャップと組み合わせやすいこと。
- QMK/Remap対応により、キーマップ調整やカスタマイズがしやすいこと。
- ホットスワップ対応により、スイッチ交換の自由度があること。

### 製品ページでの見せ方

単なる仕様表ではなく、以下のような見出しで魅力を伝える。

- 標準キーキャップで楽しめる60%格子配列
- TrackPointで完結する入力環境
- GH60互換ケースで広がるカスタマイズ
- QMK/Remap対応で自分の使い方に合わせる
- ビルドガイドとFirmwareをGitHubで確認できる

## 操作可視化コンテンツの検討

### 結論

操作可視化コンテンツは、Techmech keysのサイトコンセプトを邪魔しない。むしろ、OLSK60 v2のようにTrackPoint、レイヤー、QMK/Remap対応などの特徴がある製品では、静止画や仕様表だけでは伝わりにくい操作感を補足できるため、製品理解と購入判断を助けるコンテンツになる。

ただし、トップページの主役はあくまで製品の魅力、写真、購入・サポート導線にする。操作可視化はファーストビューに大きく置くのではなく、製品詳細ページやサポート導線の近くに配置するのがよい。

### 組み込む目的

操作可視化は、以下を伝えるために使う。

- TrackPoint搭載によって、キーボードから手を離さずに操作できること。
- レイヤー切り替えや特殊キーの位置が直感的に理解できること。
- QMK/Remap対応により、使い方に合わせてキーマップを調整できること。
- ビルド後にどのような操作体験になるかを購入前にイメージできること。

### 想定する表現

初期段階では、実装負荷が低く、サイト全体の雰囲気を壊しにくい表現から始める。

| 表現 | 内容 | 初期採用 | 備考 |
| --- | --- | --- | --- |
| 静止図 | キー配列、レイヤー、TrackPoint位置を図で示す。 | 推奨 | 軽量で保守しやすい。 |
| 短いGIF/動画 | TrackPoint操作やレイヤー切り替えを数秒で見せる。 | 推奨 | 製品の魅力が伝わりやすいが、容量に注意する。 |
| インタラクティブ配列図 | キーにホバー/タップすると割り当てを表示する。 | Phase 2以降 | 実装コストが上がるため初期は必須ではない。 |
| Remap/QMKへの導線 | 実際の設定変更は外部ツールやGitHubへ誘導する。 | 推奨 | サイト内で設定機能を持たない。 |

### 配置場所

操作可視化は、以下の位置に置くとサイトコンセプトを邪魔しにくい。

1. 製品詳細ページの「特徴」セクション内
2. OLSK60 v2の「TrackPointで完結する入力環境」見出しの直後
3. 「QMK/Remap対応」説明の補足として、レイヤーやキー割り当て例を表示
4. サポートページの「操作ガイド」導線の近く

トップページには、操作可視化を大きく載せすぎない。トップページでは代表画像と短い説明を優先し、必要であれば「操作イメージを見る」ボタンで製品詳細ページへ誘導する。

### サイトコンセプトへの影響

操作可視化は、以下の条件を守ればサイトコンセプトを強化する。

- 見た目を技術デモに寄せすぎず、製品写真やブランドトーンと合わせる。
- アニメーションは短く、控えめにする。
- 自動再生する場合は音を使わず、ページ表示速度を損なわない容量にする。
- 操作可視化だけで完結させず、購入リンクやサポートリンクへつなげる。
- 正確なキーマップやFirmware情報はGitHub側を正とし、Webサイトでは概要と入口に留める。

### 初期実装案

初期リリースでは、以下の順で進める。

1. OLSK60 v2のキー配列・TrackPoint位置を示す静止図を作る。
2. 製品詳細ページに「操作イメージ」セクションを追加する。
3. TrackPoint操作、レイヤー切り替え、Remap対応を短い説明文で補足する。
4. 詳細な操作ガイドはGitHubへリンクする。
5. 余裕があれば、短いGIFまたは動画を追加する。

### 必要な素材

- OLSK60 v2の正確なキー配列図
- TrackPoint位置が分かる画像または図
- デフォルトキーマップ情報
- レイヤー切り替え例
- 操作ガイドURL
- GIF/動画を使う場合は、軽量な操作デモ素材

## デザインとコンテンツの分離管理

デザインとコンテンツは分けて管理できる。初期段階では、見た目の実装より先に、サイト構成、製品情報、サポート導線、購入導線をMarkdown/JSONとして整理する。後からAstroなどのテンプレート側でこのコンテンツを読み込み、カード、製品ページ、サポートページとして表示する。

この進め方の利点は以下である。

- デザイン未確定でも、掲載内容のレビューを先に進められる。
- 製品名、説明、購入リンク、Firmwareリンクなどの差分が追いやすい。
- デザイン変更時に本文やリンクを壊しにくい。
- 将来、商品数が増えた場合も同じ型で追加しやすい。

初期コンテンツは以下に配置する。

| 種別 | 配置 | 用途 |
| --- | --- | --- |
| トップページ原稿 | `content/pages/home.md` | ヒーロー、ブランド紹介、購入・サポート導線の原稿 |
| 製品情報 | `content/products/olsk60-v2.md` | OLSK60 v2の製品紹介、特徴、リンク、操作可視化方針 |
| サポート原稿 | `content/support/olsk60-v2.md` | OLSK60 v2のサポート導線 |
| ナビゲーション | `content/site/navigation.json` | グローバルナビ、外部リンク |

実装段階では、Astroの`src/content/`へ移すか、現在の`content/`を読み込む形に調整する。現時点ではデザインに依存しない原稿管理を優先する。

## コンテンツデータとして管理する項目

製品情報は、ページに直接書くよりもデータとして分離する。初期はMarkdownまたはTypeScript/JSONのデータファイルで管理する。

製品データの項目案は以下である。

| 項目 | 内容 |
| --- | --- |
| `name` | 製品名 |
| `slug` | URL用識別子 |
| `summary` | 一覧カード用の短い説明 |
| `description` | 詳細説明 |
| `status` | 試作品、販売中、在庫なし、準備中など |
| `heroImage` | メイン画像 |
| `features` | 主要特徴 |
| `specs` | 仕様 |
| `shopUrl` | BOOTHなどの購入リンク |
| `supportUrl` | GitHubサポートページ |
| `firmwareUrl` | Firmware配布先 |
| `guideUrl` | ビルドガイド |
| `notes` | 注意事項 |
